# Final Whistle 文档 Review

## 结论

现有 PRD 和技术方案的产品方向是成立的，v1 范围也基本克制，适合做一个几天内可交付的闭环产品。

但两份文档目前还不适合作为“直接开工且不返工”的唯一依据，原因不是方向错误，而是存在若干关键实施歧义。如果不先收口，开发阶段会在认证、数据来源、接口定义、聚合规则和页面范围上反复调整。

建议结论：

- 保留现有产品定位与主流程
- 在开工前补齐 v1 spec
- 以“种子数据优先、认证简化、聚合实时、页面克制”为落地原则

---

## 高优先级问题

### 1. 认证方案没有真正定稿

现状：

- PRD 写“优先 OAuth”
- 技术方案写 `POST /auth/login`，示例却是 `provider=dev`
- 技术方案同时写了 `JWT 或 HTTP-only Cookie`

问题：

- 这会直接影响前后端登录流、接口安全、跨域、部署方式和本地开发复杂度
- 如果先按 JWT 做，后面再切 Cookie，前端 API client 和 middleware 会返工

建议：

- v1 明确采用 `HTTP-only Cookie Session`
- v1 登录能力明确为 `dev login + GitHub OAuth 二选一`
- 如果目标是“几天内完成闭环”，建议先做 `dev login`
- OAuth 放到 v1.1，不要卡住主流程

决策建议：

- `v1`: dev login + cookie session
- `v1.1`: GitHub OAuth

---

### 2. 比赛、球队、球员的数据来源没有定稿

现状：

- PRD 提到“先做 seed data”
- 技术方案保留了大量比赛、球队、球员页面与接口

问题：

- 如果没有先定义“数据只来自内部 seed”，后端模型、分页、筛选、唯一键、时间字段和同步策略都不稳定
- 如果中途引入外部足球数据 API，项目复杂度会立刻上升一个等级

建议：

- v1 明确只使用内部 seed data
- 只覆盖一个赛季、少量球队、少量比赛、少量球员
- 所有实体使用内部主键；如需外部映射，预留 `external_source` / `external_id` 字段，但不启用同步能力

---

### 3. PRD 与技术文档对前后端职责存在不一致

现状：

- PRD 的系统交互视图里还出现了 `Server Action: Create CheckIn`
- 技术方案定义的是前后端分离，Next.js 通过 HTTP 调 Gin API

问题：

- 这会误导实现方式，尤其是在前端提交表单、鉴权和错误处理上

建议：

- 删掉 PRD 中的 `Server Action` 表述
- 统一写法为：Next.js 调用 Gin API

---

### 4. 评分与聚合规则不够明确

现状：

- 文档定义了比赛评分、球队评分、球员评分
- 但没有定义缺失值、样本过少、编辑后重算、是否展示小数位等规则

问题：

- 这些不是 UI 细节，而是接口契约的一部分
- 如果不先定，前端展示和后端查询会各自假设

建议至少补齐：

- 评分范围固定为 `1-10` 整数
- 球员评分最多 `5` 人，最少 `0` 人
- 平均分保留 `1` 位小数
- 聚合仅统计有效 check-in
- 用户编辑 check-in 后，聚合按最新数据实时重算
- 样本数 `< 3` 时前端展示“样本较少”

---

### 5. 标签系统缺少约束定义

现状：

- PRD 有建议标签
- 技术方案只有 `Tag` / `CheckInTag` 模型

问题：

- 没定义标签来源、是否固定、展示顺序、是否允许下线

建议：

- v1 采用固定标签字典
- 标签由 seed 初始化
- 标签字段至少包括：`id`, `name`, `slug`, `sort_order`, `is_active`
- 前端只读取启用标签，不支持用户自建标签

---

## 中优先级问题

### 6. 页面范围略多，容易稀释主流程

现状：

- v1 同时包含首页、比赛列表、比赛详情、球队页、球员页、个人主页、登录页

风险：

- 对一个几天内交付的项目来说，球队页和球员页很容易做成“有入口但价值不高”的占位页

建议：

- 保留球队页、球员页，但明确为简版只读页
- 如果时间不足，优先保证比赛详情页和个人主页完整
- 首页可以直接复用比赛列表，不单独设计新信息架构

---

### 7. 列表筛选能力与 seed data 规模可能不匹配

现状：

- PRD 要求赛事筛选、赛季筛选、排序、可选搜索球队名

问题：

- 如果 v1 只有少量 seed data，这些功能的用户价值有限

建议：

- v1 最小可交付只保留：
  - 按赛季筛选
  - 按赛事筛选
  - 按比赛时间倒序
- `keyword` 搜索作为可选项，不进入必须里程碑

---

### 8. 用户主页统计项还缺一层定义

现状：

- PRD 要“常打卡球队 / 常评分球员”
- 技术文档提供了接口名字，但没定义计算规则

建议：

- 常打卡球队：按 check-in 次数排序，取前 3
- 常评分球员：按用户打过分的球员出现次数排序，取前 3
- 并列时按最近一次评分时间倒序

---

## 低优先级问题

### 9. 文档中的接口细节还缺分页统一规范

建议统一：

- 列表接口返回 `items`, `page`, `pageSize`, `total`
- 默认 `page=1`
- 默认 `pageSize=20`
- 最大 `pageSize=50`

### 10. 命名风格建议统一

建议统一为：

- API 字段：`camelCase`
- 数据库表：`snake_case`
- 枚举值：全大写，例如 `FULL`, `PARTIAL`, `HIGHLIGHTS`

---

## 评审后的推荐收口版本

建议把 v1 真正收口为下面这条线：

1. 使用 seed data 提供比赛、球队、球员数据
2. 用户通过简化登录进入系统
3. 用户在比赛详情页创建或编辑唯一一条 check-in
4. 系统实时展示比赛聚合评分与最近短评
5. 用户在个人主页查看自己的观赛档案

这条线是闭环的，而且工程复杂度可控。

---

## 最终建议

可以继续推进，不需要推翻重写。

但必须先补三类文档后再开工：

- 模块化开发计划
- 明确的里程碑与验收口径
- 一份真正可执行的 v1 spec
