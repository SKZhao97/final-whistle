# Final Whistle 数据源方案草案

## 1. 目标

在**不投入金钱成本**的前提下，为 Final Whistle 后续接入真实足球数据提供一条可持续、可演进的方案。

核心原则：

- 免费优先
- API 优先
- 手工导入兜底
- 爬虫只做补充，不做主链路

---

## 2. 方案结论

后续真实数据接入采用三层方案：

1. **主方案：免费 API**
2. **兜底方案：手工导入**
3. **补充方案：轻量爬虫**

不建议将爬虫作为核心数据来源。

---

## 3. 为什么不用爬虫做主方案

爬虫存在以下问题：

- 目标站点页面结构容易变化，维护成本高
- 可能存在反爬、限流、验证码、Cloudflare 等问题
- 字段清洗和去重成本高
- 长期稳定性差
- 合规风险通常高于 API

对于 Final Whistle 这种以“赛后记录”为核心的产品，数据源最重要的是：

- 比赛、球队、球员结构稳定
- 比赛结果和状态可靠
- 能持续补充新比赛

因此主链路更适合建立在结构化 API 上。

---

## 4. 推荐架构

### 4.1 数据接入分层

建议分成三层：

#### A. 采集层

负责从外部来源获取原始数据：

- 免费足球 API
- 本地 JSON / CSV 导入文件
- 必要时的小型补充爬虫

#### B. 标准化层

负责将外部数据映射到 Final Whistle 自己的领域模型：

- Team
- Player
- Match
- MatchPlayer

这层的目标是让外部数据源发生变化时，不直接影响产品逻辑。

#### C. 产品层

前端页面、CheckIn、聚合逻辑只依赖本地数据库中的标准化数据，不直接依赖第三方 API 响应。

---

## 5. 数据来源策略

### 5.1 主方案：免费 API

免费 API 优先承担这些核心数据：

- competition
- season
- teams
- players
- matches
- kickoff time
- status
- final score
- 参赛球员 / lineup / participants

目标不是一开始覆盖所有联赛，而是先覆盖：

- 少量核心赛事
- 少量目标赛季
- 当前产品真正需要展示的比赛集

### 5.2 兜底方案：手工导入

如果免费 API 存在覆盖不足、字段缺失、质量不稳定的问题，可以通过本地导入文件补齐。

手工导入适合补这些数据：

- 少量比赛
- 球队 logo
- short name / slug
- venue
- 某些球员资料
- API 缺失或错误字段

这类方式虽然不自动，但几乎零成本，而且非常可控。

### 5.3 补充方案：轻量爬虫

仅在以下情况下考虑爬虫：

- 免费 API 没有某个非核心字段
- 手工维护该字段过于重复
- 该字段不会影响核心链路稳定性

适合被爬虫补充的内容：

- logo
- venue
- round 文本
- 个别补充资料

不建议用爬虫承担：

- 比赛主数据
- 比赛状态与比分主数据
- 球员主数据主链路

---

## 6. 对现有模型的建议

虽然当前 v1 不实现外部同步，但为了后续扩展，建议在接入真实数据时为核心实体补充：

- `external_source`
- `external_id`

优先考虑这些实体：

- Team
- Player
- Match

这样未来即使切换免费 API，或者同时接多个来源，也能通过映射层维持本地数据稳定。

---

## 7. 建议的实施顺序

### Phase A

继续使用当前 seed data，完成产品主流程开发。

### Phase B

接入一个免费 API，先做一次性导入脚本，不做复杂自动同步。

### Phase C

在免费 API 覆盖不足的地方，引入手工导入补齐。

### Phase D

只有在确实值得自动化时，再补轻量爬虫。

---

## 8. 设计原则

- 不让前端直接依赖第三方 API
- 不让核心产品逻辑依赖爬虫
- 不为了“全自动”牺牲稳定性
- 先解决“够用”，再解决“自动化”
- 数据源可以替换，但本地领域模型保持稳定

---

## 9. 一句话结论

Final Whistle 后续真实数据接入应采用：

**免费 API 做主干，手工导入做兜底，轻量爬虫做补充。**
